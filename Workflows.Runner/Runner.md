### Why the Upward Traversal?

The "Upward Traversal" (or bubbling) is essential because of **Nested Composition**. In a complex workflow, a task isn't just a flat line; it’s often part of a hierarchy (e.g., a Task inside a "Wait All" Group, which is inside a "Sequence").

When a leaf-level task completes:
1.  **Status Propagation:** It tells its parent, "I’m done."
2.  **Logic Evaluation:** The parent must then check its own rules. If it’s a "Wait All" group, it checks if it’s the *last* child to finish.
3.  **Continuation:** If the parent is now satisfied, it too becomes "Completed," and the engine must move to the parent's sibling or the next step in the outer scope.

Without the upward traversal, the engine would complete the small task but the "Parent" would stay stuck in a waiting state forever.

### Documentation: Implementing `WorkflowRunner`

This documentation outlines how to implement the `WorkflowRunner` logic using an in-memory execution model where the final state is handed off to an external service.

#### 1. The Core Execution Loop (`RunWorkflow`)
The runner must act as a "state compressor." It takes the current state, applies an external signal (the event), and runs the C# enumerator until it hits the next logical pause point or the end of the method.



#### 2. Component Responsibilities
* **`matchExpressionTransformer`**: Used to evaluate if the `SignalDto` actually matches the requirements of the `IncomingWait` (e.g., matching an OrderID).
* **`IAsyncEnumerator<Wait>`**: This is your `FunctionRunner`. It wraps the user's workflow method.
* **`WorkflowStateDto`**: This contains the serialized local variables and the "Pointer" (the state of the enumerator).

---

#### 3. Step-by-Step Implementation Logic

##### Step A: De-serialization & Setup
1.  Use the `objectSerializer` to restore the `WorkflowState` into a live object instance.
2.  Initialize your `IAsyncEnumerator` (the Runner) with this state. 
3.  **Crucial:** You must manually advance the state to where it last "yielded."

##### Step B: The "Bubbling" Logic
To replicate the `ResumeExecution` logic without a DB:
1.  **Match Check:** Verify the `Signal` matches the `IncomingWait`.
2.  **Completion:** Mark the `IncomingWait` as completed.
3.  **Traverse Up:**
    * If the current wait is a child of a **Group**, check if the group's conditions are met (e.g., all children are now completed).
    * If the group is satisfied, mark the **Group** as completed and continue the loop upward.
    * If the group is **not** satisfied, stop and return the current state.

##### Step C: Advancing the Code (`MoveNextAsync`)
Once a "Wait" or a "Wait Group" is fully satisfied:
1.  Call `await enumerator.MoveNextAsync()`.
2.  **If `true`**: The code hit a new `yield return`. Capture the new `Wait` object. This becomes your `IncomingWait` for the next run.
3.  **If `false`**: The workflow reached the end. Set `Status = Completed`.

---

#### 4. The Result Object
Since you aren't saving to a DB, your `WorkflowRunResult` must act as the "Deltas" for the calling service.

| Property | Value to Populate |
| :--- | :--- |
| **`WorkflowState`** | The serialized version of the object after `MoveNextAsync` stopped. |
| **`IncomingWait`** | The *new* wait entity generated by the code. |
| **`Status`** | `InProgress`, `Completed`, or `InError`. |

---

#### 5. Handling Errors
Wrap the `MoveNextAsync` call in a `try-catch`. 
* If an exception occurs inside the user's workflow code, capture the stack trace and message.
* Set the `WorkflowRunResult.Status` to `InError`.
* **Important:** Do not lose the `WorkflowState`, as the user might want to fix the data and "Retry" from the exact same point.

---

### Implementation Tip: The "Pseudo-Recursive" Loop
In your `RunWorkflow` method, use a pattern similar to this:

```csharp
var currentWait = context.WorkflowState.CurrentWait;
bool canMoveForward = CheckCompletion(currentWait, context.Signal);

while (canMoveForward) {
    var parent = currentWait.Parent;
    if (parent is GroupWait group && !group.IsSatisfied()) {
        break; // Stop bubbling, group needs more signals
    }
    
    // If we reach here, we can actually execute code
    var hasMoreCode = await enumerator.MoveNextAsync();
    if (!hasMoreCode) {
        result.Status = Completed;
        break;
    }
    
    currentWait = enumerator.Current; // The new yield point
    canMoveForward = false; // Stop until the next signal comes in
}
```

This ensures your runner is clean, stateless, and perfectly suited for a "Side-Effect-Free" execution model where the caller handles the final persistence.
